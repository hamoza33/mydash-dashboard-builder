# Self-contained HTML dashboard via gzip+base64 data embedding

The dashboard builder was gutted and rewritten to produce a single self-contained HTML file with all rendering logic inline. Instead of accepting pre-computed `MetricsResult` and server-side city performance data, the new interface takes raw reconciliation rows (45-column `string[][]`), transforms them to a 34-field JSON format, gzip-compresses and base64-encodes the result as a `PAY` constant, and wraps everything in ~35KB of minified CSS + client-side JS that renders 14+ charts, two tabs, filters, and a detail modal. The pipeline worker was updated to generate the slug before calling the builder (the slug is embedded into `window.DASHBOARD_DATA_API`) and to pass rows directly.

**Watch for:** the `carrier` field is always empty string — the builder never extracts it from any column, so Carrier Performance charts render with "No Data" labels only (confirmed). The `sheetUrl` is interpolated directly into a `<script>` block without escaping or validation — the old `sanitizeUrl` was removed — creating an XSS vector if the URL contains quote characters (confirmed). No test coverage exists for the new builder at all.

## High-level view

The architecture flipped from server-computes-everything to server-ships-raw-data-and-client-renders. Every analytics computation now lives in a ~5KB blob of minified client-side JS that has zero test coverage and is difficult to unit-test in its current string-literal form.

The field mapping from the 45-column reconciliation format to the 34-field dashboard JSON leaves several fields permanently blank (`carrier`, `origin_city`, `dest_city`, `lf_date`, `funnel_url`, `utm_medium`, `status_detail`, `latest_update`). Most are informational gaps, but `carrier` is actively used by the Carrier Performance chart, meaning that chart is structurally broken.

The worker restructuring moved slug generation above `buildDashboardHtml` so the slug can be baked into the HTML. When `importClient` is null (no IMPORT_MCP_URL configured) or when `list_dashboards` throws, the slug never gets deduplicated, so a repeated run for the same product+date silently overwrites the previous deployment.

The removal of `sanitizeUrl` and the direct interpolation of `sheetUrl` into a `<script>` tag is a regression from the prior version's security posture.

## Carrier field is structurally unpopulated

The `transformRow` function assigns `carrier: ""` unconditionally:

```typescript
carrier: "",
```

The client-side `dual('car', s, 'carrier', 99, 'dr', 1)` call groups all rows by this field and renders a multi-bar chart. With every row having an empty carrier, the chart shows a single "No Data" bar with the aggregate delivery/return rates — functionally useless.

The 45-column format has "Shipping Method" at index 21 (column V). Whether that maps cleanly to carrier depends on what the data sources put there, but it's the only candidate. Alternatively, many tracking number formats encode the carrier as a prefix (e.g., SAL, SMF, ARAMEX), which could be extracted with a lookup.

## XSS regression from sanitizeUrl removal

The new builder interpolates `sheetUrl` directly into a JavaScript string literal inside a `<script>` tag:

```typescript
window.DASHBOARD_SHEET_URL="${safeSheetUrl}";
```

Despite the variable name `safeSheetUrl`, no sanitization is applied — it's just `sheetUrl || ""`. The `sheetUrl` comes from MCP tool results in the worker (`res.url as string`), so it's controlled by an upstream service rather than direct user input. The risk is moderate (requires a compromised MCP response), but the fix is trivial: use `JSON.stringify(sheetUrl || "")` to produce a safely-escaped JS string literal, or restore URL validation.

## No size guard for growing datasets

As row counts grow beyond the current 431 rows, the base64-encoded payload scales roughly linearly. At ~80 bytes per compressed row (rough estimate from 35KB / 431 rows), a 3000-row dataset would push ~240KB of payload alone, blowing past the 60KB limit. The builder does not check output size or truncate. This is a latent failure mode that will surface when product runs accumulate more leads.

## Slug collision on list_dashboards failure

When `importClient` exists but `list_dashboards` throws, the catch block logs a warning and proceeds with the base slug. If a dashboard with that slug already exists, the deploy call overwrites it silently. The fail-open behavior here means a transient MCP error can destroy a previous dashboard's content.

## Client-side JS as untestable string literal

The rendering engine is a single `getClientJS()` string returned from a TypeScript function — no syntax checking at build time, no way to unit-test individual chart functions without a full browser environment (jsdom lacks `DecompressionStream`). A typo in the minified JS would only surface at runtime in the deployed dashboard. The `normalizeStatus` logic is baked at build time during `transformRow`, so the client reads pre-normalized values — but this also means normalization rules can't be updated without regenerating the HTML.
