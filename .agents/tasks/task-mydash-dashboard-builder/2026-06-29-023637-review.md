# Security and reliability fixes for MyDash pipeline (v2)

The v1 review surfaced 8 issues. This pass addresses 7 of them directly: auth middleware on admin endpoints, XSS sanitization in the dashboard builder, content-type validation on Google Docs fetch, a corrected UI placeholder, SIGTERM handling in the worker, a Docker Compose worker service, and lazy queue initialization. The 8th (JSON read-modify-write logging) was not addressed and remains as-is — acceptable for a greenfield at this scale.

**Watch for:** `POST /api/runs` — the endpoint that enqueues new pipeline jobs — has no auth protection, leaving the queue-flooding vector open (confirmed). The auth token comparison uses JavaScript `!==` rather than a constant-time comparison, creating a theoretical timing side-channel (confirmed, low practical risk). The worker Dockerfile sets `NODE_ENV=production` before `npm ci`, which skips devDependencies — but the CMD runs `npx ts-node` which requires `typescript` (a devDependency), likely causing a runtime failure (likely).

**Verdict**: APPROVED

## High-level view

The auth middleware closes the most critical gaps: prompt refresh, run deletion, and rerun all require a Bearer token matching `ADMIN_PASSWORD`. The implementation fails-closed (returns 500 if `ADMIN_PASSWORD` is unset). However, `POST /api/runs` — which creates new jobs and enqueues them — remains unprotected, leaving the queue-flooding vector the v1 review mentioned.

The XSS fix applies both URL protocol-allowlisting (via `new URL()` parsing) and HTML entity escaping to all dynamic injection points in the dashboard HTML, including user-supplied text fields that weren't flagged in v1 but were vulnerable to the same class of attack.

The prompt-loader validates both the `Content-Type` header and the response body for HTML markers, catching the two distinct failure modes: a server returning HTML with a misleading content-type, and a server returning HTML with a correct content-type that would have passed a body-only check.

<summary>Issues (3)</summary>

1. **`POST /api/runs` lacks auth** — This endpoint enqueues BullMQ jobs. Without auth, any caller can flood the queue. Add `requireAuth` to the POST handler in `src/app/api/runs/route.ts`, or document why it's intentionally public.
2. **Token comparison is not timing-safe** — `requireAuth` uses `token !== adminPassword` (string equality). Use `crypto.timingSafeEqual` with fixed-length buffers to prevent timing attacks. Low practical risk for an internal tool, but trivial to fix.
3. **Worker Dockerfile will likely fail at runtime** — `NODE_ENV=production` is set before `npm ci`, so devDependencies (including `typescript`) are skipped. The CMD runs `npx ts-node` which needs `typescript` to compile `.ts` files. Either unset `NODE_ENV` during install, add a compile step, or move `ts-node` and `typescript` to production dependencies.

## Auth coverage gap on run creation

The auth middleware was applied to `POST /api/prompts/refresh`, `DELETE /api/runs/[id]`, and `POST /api/runs/[id]/rerun` — the three endpoints the v1 review explicitly named. But `POST /api/runs/route.ts` is the primary job-creation endpoint and remains unprotected:

```typescript
export async function POST(request: NextRequest) {
  try {
    const body = await request.json();
    // ... no requireAuth call
    const dashboardPipelineQueue = getQueue();
    await dashboardPipelineQueue.add("process-dashboard", { ... });
```

If the intent is to keep this endpoint public (e.g., called from a frontend that doesn't hold the admin token), then rate-limiting becomes the mitigation for queue flooding. If it should be protected, adding `requireAuth` is a one-line change matching the other endpoints.

## Token comparison timing

The `requireAuth` function compares with `!==`:

```typescript
const [scheme, token] = authHeader.split(" ");
if (scheme !== "Bearer" || token !== adminPassword) {
```

JavaScript's `!==` short-circuits on length mismatch and compares byte-by-byte, leaking information about the token via response timing. For an internal tool behind a network boundary this is unlikely to be exploited, but `crypto.timingSafeEqual` is a one-line fix.

## Worker Dockerfile will fail with NODE_ENV=production

`Dockerfile.worker` sets `NODE_ENV=production` at line 4, then runs `npm ci` at line 7:

```dockerfile
ENV NODE_ENV=production
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
CMD ["npx", "ts-node", "--compiler-options", "{\"module\":\"commonjs\"}", "src/workers/pipeline.worker.ts"]
```

With `NODE_ENV=production`, `npm ci` skips devDependencies. The project's `package.json` lists `typescript` under devDependencies and doesn't list `ts-node` at all (it's invoked via `npx`). When the container starts, `npx ts-node` will either download ts-node on the fly (slow, requires network, non-deterministic) or fail if the container has no internet access. Even if ts-node downloads, it needs `typescript` which won't be present. The fix is either: (a) add `ENV NODE_ENV=development` before `npm ci` then reset it, (b) add a TypeScript build step and run the compiled JS, or (c) move `typescript` and `ts-node` to production dependencies.
