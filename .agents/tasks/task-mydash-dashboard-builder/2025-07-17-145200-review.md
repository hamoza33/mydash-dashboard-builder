# MyDash Dashboard Builder: full-stack pipeline for e-commerce reconciliation

A greenfield Next.js application that orchestrates an e-commerce data reconciliation pipeline. The system loads prompts from published Google Docs, fetches order/lead/tracking data from multiple MCP-connected services, runs a reconciliation engine (phone normalization, schema detection, city mapping, 45-column row assembly), and generates standalone HTML dashboards. Infrastructure is BullMQ on Redis for background processing, Prisma/SQLite for persistence, and Docker Compose for deployment.

**Watch for:** The admin endpoints (prompt refresh, run delete) are completely unprotected — no auth check of any kind (confirmed). The `sheetUrl` from MCP responses is rendered directly into dashboard HTML `href` attributes without sanitization, opening an XSS vector (confirmed). BullMQ queue instantiation at import time means every API route that touches `@/lib/queue` will fail-open with unhandled Redis connection errors during cold starts if Redis is down (likely). The `defaultCc` form placeholder says "e.g. MA" but the engine expects numeric codes like "966" — users will produce garbage phone numbers (confirmed).

## High-level view

The architecture separates API routes (CRUD + job enqueue), a BullMQ worker (multi-step pipeline), and the reconciliation engine (pure functions, no I/O). The reconciliation modules are independently testable and covered by 69 unit tests, but none of the I/O layer (worker, API routes, prompt loader, MCP client, dashboard builder) is tested.

The security posture is the weakest layer. There is no authentication on any endpoint. The `ADMIN_PASSWORD` env var exists but is never checked. The prompt refresh endpoint and run deletion are world-accessible, and the dashboard HTML builder injects external URLs without escaping.

The prompt-loading design fetches Google Docs via `/export?format=txt`, hashes content for change detection, and stores versioned snapshots. It will silently succeed with garbage content (an HTML login page returned as HTTP 200) if doc permissions change, since there's no content-type validation.

The pipeline worker's error handling is all-or-nothing: any MCP failure at any step marks the entire run as failed. No partial recovery, no step-level retry, and a transient failure in step 6 of 9 discards all previously fetched data. The worker also lacks SIGTERM handling for graceful container shutdown.

<summary>Issues (8)</summary>

1. **No authentication on admin/mutation endpoints** — Add auth middleware (at minimum check `ADMIN_PASSWORD` header) to POST `/api/prompts/refresh`, DELETE `/api/runs/[id]`, and POST `/api/runs/[id]/rerun`.
2. **XSS via unsanitized `sheetUrl` in dashboard HTML** — The `buildDashboardHtml` function injects `sheetUrl` directly into an `<a href>`. Validate it's a well-formed HTTPS URL or HTML-escape it before embedding.
3. **Google Docs fetch lacks content-type validation** — If a doc becomes private, the export URL returns a login page (HTML) as a 200 response. Check `Content-Type: text/plain` or validate the response doesn't start with `<!DOCTYPE`.
4. **Pipeline has no partial recovery** — A failure in step 6 discards steps 1-5 data. Consider persisting intermediate results to the run record so reruns can skip already-fetched steps.
5. **BullMQ queue instantiation at import time** — `dashboardPipelineQueue` is created when `queue.ts` is imported, triggering Redis connections during cold starts of any API route that imports the module. Defer queue creation or catch connection errors.
6. **Worker lacks SIGTERM handling** — In container environments, the BullMQ worker won't gracefully finish in-progress jobs on shutdown. Add signal handlers that call `worker.close()`.
7. **`defaultCc` field ambiguity** — The form placeholder says "e.g. MA" (country abbreviation) but the reconciliation engine expects a numeric country code like "966". The UI and the backend disagree on semantics.
8. **Logs stored as JSON string column with read-modify-write** — Each `appendLog` call reads the full JSON string, parses, pushes, re-serializes, and writes back. This degrades with log volume and has no concurrency protection.

<summary>Details</summary>

## No authentication on any endpoint

Every API route is publicly accessible. The `ADMIN_PASSWORD` env var exists in `.env.example` but is never checked anywhere in the codebase:

- `POST /api/prompts/refresh` triggers outbound HTTP requests to Google Docs and writes to the database
- `DELETE /api/runs/[id]` permanently deletes run records
- `POST /api/runs` enqueues BullMQ jobs (potential for queue flooding)
- `POST /api/runs/[id]/rerun` clones and re-enqueues jobs

No rate limiting, no API key check, no session validation. Any caller can trigger pipeline runs, delete data, or exhaust the Redis queue.

## XSS vector in dashboard HTML generation

In `dashboard-builder.ts`, the `sheetUrl` variable is injected directly into HTML:

```typescript
<a href="${sheetUrl}" target="_blank" class="sheet-link">Open Sheet</a>
```

The `sheetUrl` comes from an MCP tool response (`sheetResult.url`). If a compromised or buggy MCP endpoint returns a `javascript:` URL or contains quotes that break out of the attribute, the generated HTML becomes an XSS payload. Since these dashboards are deployed and served to end users, this is a real attack surface.

The `productName`, `dateFrom`, and `dateTo` values also appear unescaped in the HTML `<title>` and `<h1>` tags, though these originate from the user's form submission rather than external services.

## Google Docs prompt fetch assumes public access permanently

`PromptLoader.fetchDocContent` checks `response.ok` but doesn't validate the content type. Google's export endpoint returns HTTP 200 with an HTML login page when a document is no longer publicly shared. The system would store an HTML login page as the "prompt content", hash it, version it, and inject it into the pipeline — causing confusing downstream failures rather than a clear error at fetch.

## Pipeline error model: all-or-nothing with read-modify-write logging

The pipeline worker uses a single try/catch around all 9 steps. Any failure marks the run as `failed` with no indication of which steps completed or what data was already fetched. BullMQ's retry mechanism re-runs the entire pipeline from step 1, re-fetching data that was already successfully retrieved.

The logging pattern compounds this: `appendLog` does a `findUnique` + JSON.parse + push + JSON.stringify + `update` on every log entry. With ~15-20 log entries per run, this means 15-20 sequential read-modify-write cycles against the same row with no locking.

## Queue instantiation side effects

`src/lib/queue.ts` exports `dashboardPipelineQueue` as a module-level constant. Any file that imports from this module — including API routes that enqueue jobs — will open a Redis connection at import time. If Redis is unavailable during a Next.js cold start, the underlying ioredis connection will emit `error` events that are not caught anywhere in `queue.ts`. This fails open: the route handler starts executing but will throw when it tries to `add()` a job.

## `defaultCc` semantic mismatch between UI and engine

The `VariablesForm` component labels the field "Default CC" with placeholder "e.g. MA". The reconciliation engine's `normalizePhone` function uses `defaultCc` as a numeric prefix (e.g., "966" for Saudi Arabia). If a user enters "MA" (Morocco's ISO code), phone normalization will prepend "MA" to phone numbers, producing garbage like "MA555123456". The seed data uses "966" and "971", confirming the engine expects numeric codes.

## Test coverage gaps

Not tested: the pipeline worker (`processPipeline`, `updateRunStatus`, `appendLog`, `callMcpTool`), no tests for any API route, no tests for the prompt loader, no tests for the dashboard HTML builder, no tests for the MCP client. The untested surface is where all the I/O and side effects live — and where the content-type validation gap, the auth gap, and the XSS vector all reside.

## Docker deployment gap

The Dockerfile copies from `.next/standalone` but the Docker Compose file doesn't include the BullMQ worker as a separate service. In production, the worker process (`npm run worker`) needs to run alongside the app, either as a second container or via a process manager, but the compose file only defines the `app` service.
